Building a Real-World Computer Vision System for Heritage Conservation

From dataset to deployment: how I built a full-stack AI system to automate heritage monitoring.

1. The Real-World Problem

Pushkarnis (ancient temple tanks) are vital to our heritage, but they're rapidly degrading from pollution and structural damage. Manual inspection is slow, inconsistent, and doesn't scale. The core challenge I tackled was: can we automate heritage monitoring using computer vision to make conservation efforts more effective?

2. A Full-Stack System, Not Just a Model

I built a complete end-to-end pipeline, not just an isolated ML model. The system includes:

  • Image input via a user-facing web interface.
  • A robust preprocessing pipeline to handle real-world data.
  • Multiple deep learning models for classification and analysis.
  • A prediction layer that delivers actionable insights.
  • A Flask-based backend to tie everything together.

3. The Unseen Work: Dataset Engineering

This was the most critical phase. I collected and annotated over 2600+ real-world images, which came with challenges like lighting variations, environmental noise, and seasonal changes. I applied rigorous normalization, augmentation, and standardization techniques to build a robust dataset—a step many academic projects overlook.

4. Exploring Architectural Tradeoffs

Instead of settling on one model, I experimented with several architectures (MobileNet, ResNet50, DenseNet, EfficientNet, and even the advanced DINOv2) to understand the real-world tradeoffs between accuracy, inference speed, and computational cost.

5. What The System Actually Does

This is more than a classifier. It's a multi-module intelligent system that performs:

  • Architectural style classification.
  • Crack and erosion detection.
  • Water quality estimation based on visual cues.
  • Real-time predictions via the web interface.

6. Honest Engineering Challenges

The journey wasn't simple. I navigated dataset imbalance, managed overfitting in deeper models, and constantly balanced the accuracy vs. inference speed tradeoff, especially when considering deployment on less powerful hardware.

7. Key Engineering Insight

My biggest takeaway was that model accuracy alone is not enough. For a system to be useful, deployment constraints and real-world data variability matter far more. A 99% accurate model that's too slow or fails on noisy images is useless in the field.

8. The Full-Stack Angle

This project solidified my skills as a full-stack developer. I designed the system workflow, built the Flask API backend, handled the frontend user interaction, and integrated the ML pipeline. It was a true conversion of an ML concept into a full-stack engineering product.

9. The Results

The system achieved a classification accuracy in the 96-97% range across different models, providing a reliable baseline for automated monitoring and delivering real-time predictions.

10. Future Improvements

The system is designed for growth. Future steps include integrating real-time monitoring from drone feeds, scaling the dataset further, and optimizing the models for edge deployment.

💸 Building a Blockchain-Based Transaction System Using Flask and Web3.py

A practical simulation of how blockchain transactions work in a full-stack web application using Flask and Web3.py.

1. Why I Built This

Blockchain is often explained in theory, but understanding it properly requires building something. I wanted to answer a practical question: can I simulate how blockchain transactions work using a web application? Instead of just reading about wallets, I built a system where users can send/receive transactions and maintain history through a web interface.

2. What I Built

I developed a Blockchain-Based Transaction Management System that simulates core blockchain behavior in a controlled environment. It includes user authentication, a wallet-like transaction flow, history tracking, and a backend database for persistence.

3. System Architecture

The application follows a typical full-stack structure:

  • Backend: Flask for routing, SQLAlchemy for database operations, and Web3.py to simulate blockchain interactions.
  • Frontend: HTML + CSS using Jinja templates for dynamic data rendering.
  • Database: SQLite to store users and transactions.

4. Key Features & Challenges

The system features a secure authentication system with password hashing (bcrypt) and session management (Flask-Login). Users can send and receive transactions, with a complete transaction log maintained and displayed, mimicking a blockchain ledger. The main challenges were modeling abstract blockchain concepts, designing a clean relational data model, and ensuring security even in a simulation.

5. What I Learned

This project taught me that blockchain is fundamentally about data integrity and trust systems. It reinforced that even simplified systems require clear architecture and data flow. Most importantly, building a simulation teaches far more than just reading documentation. This project gave me a practical understanding of transaction systems and strengthened my skills in backend development (Flask), database design, and system architecture.

⚠️ Important Note: This is a simulated environment built for educational purposes and does not interact with a real blockchain network.
